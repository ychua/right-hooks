# Design Doc: Configurable Skill Dispatch

Generated by /plan-ceo-review on 2026-03-22
Branch: main | Mode: SELECTIVE EXPANSION
Repo: ychua/right-hooks

## Problem Statement

The enforcement hooks (`stop-check.sh`, `pre-merge.sh`) hardcode tool detection
with `if rh_has_gstack ... elif rh_has_superpowers ... else` chains. This means:

1. Adding a third tool requires editing 3+ shell scripts
2. Users can't choose their review/QA/doc tooling — it's auto-detected
3. The "else" fallback gives generic "post a comment" messages that don't
   guide the agent to a specific action

**Failure mode without this:** Users who switch from gstack to superpowers
(or vice versa) must either re-run `init` or manually edit hook scripts.
Users with custom review tools get no guidance.

## Alternatives Considered

### A) Extend signatures.json with skill commands
- Pros: No new files, init already selects signatures
- Cons: Mixes comment pattern matching with skill dispatch — two different
  concerns in one file. User editing a skill command might accidentally break
  a regex pattern.
- **Rejected:** Separation of concerns.

### B) New `.right-hooks/skills.json` config (CHOSEN)
- Pros: Clean separation, user can edit skills without touching patterns,
  explicit fallback chain per gate
- Cons: One more config file
- **Chosen because:** Single responsibility — signatures detect comments,
  skills dispatch actions. Users customize each independently.

### C) Merge into profiles (per-branch-type skill config)
- Pros: Different branches could use different review tools
- Cons: Over-engineering — nobody varies review tooling by branch type.
  Complicates profile matching for no real user value.
- **Rejected:** Premature abstraction.

## Architecture

```
  INIT TIME                              RUNTIME (hook invocation)
  ─────────                              ────────────────────────
  detectTooling()                        hook sources _preamble.sh
       │                                       │
       ▼                                       ▼
  ┌──────────────────┐                  rh_skill_command("codeReview", PR)
  │ Copy template:   │                         │
  │ skills-gstack.json│                        ▼
  │ → .right-hooks/  │                  ┌──────────────────┐
  │   skills.json    │                  │ Read skills.json │
  └──────────────────┘                  │ (cached per hook)│
                                        └────────┬─────────┘
  USER EDITS (optional)                          │
  ─────────────────────                    ┌─────┴──────┐

    skills set review /my-review     │              │
       │                             ▼              ▼

  ┌──────────────────┐                          (with ${PR_NUM}
  │ Update           │                           interpolated)
  │ skills.json      │
  │ (atomic write)   │
  └──────────────────┘
```

## Technical Design

### skills.json Schema

Three template variants shipped in `signatures/` (alongside existing signature
files). Selected during `init` based on detected tooling.

```json
// signatures/skills-gstack.json (default when gstack detected)
{
  "codeReview": {
    "skill": "/review",
    "provider": "gstack",
    "fallback": "Dispatch a code review subagent for PR #${PR_NUM}"
  },
  "qa": {
    "skill": "/qa",
    "provider": "gstack",
    "fallback": "Dispatch a QA subagent for PR #${PR_NUM}"
  },
  "docConsistency": {
    "skill": "/document-release",
    "provider": "gstack",
    "fallback": "Post a documentation consistency comment on PR #${PR_NUM}"
  }
}
```

```json
// signatures/skills-superpowers.json
{
  "codeReview": {
    "skill": "superpowers:requesting-code-review",
    "provider": "superpowers",
    "fallback": "Dispatch a code review subagent for PR #${PR_NUM}"
  },
  "qa": {
    "skill": null,
    "provider": null,
    "fallback": "Dispatch a QA subagent for PR #${PR_NUM}"
  },
  "docConsistency": {
    "skill": null,
    "provider": null,
    "fallback": "Post a documentation consistency comment on PR #${PR_NUM}"
  }
}
```

```json
// signatures/skills-generic.json (no tools detected)
{
  "codeReview": {
    "skill": null,
    "provider": null,
    "fallback": "Post a code review comment with severity markers on PR #${PR_NUM}"
  },
  "qa": {
    "skill": null,
    "provider": null,
    "fallback": "Post a QA comment with test results on PR #${PR_NUM}"
  },
  "docConsistency": {
    "skill": null,
    "provider": null,
    "fallback": "Post a documentation consistency comment on PR #${PR_NUM}"
  }
}
```

**Fields:**
- `skill`: A runnable skill command (string or null). Must be a command the agent
  can invoke (e.g., `/review`, `superpowers:requesting-code-review`). Never prose.
  If no skill exists for this gate, set to `null`.
- `provider`: Which tool provides this skill ("gstack", "superpowers", or null).
  Used by `doctor` to validate the provider is installed.
- `fallback`: Prose text displayed when `skill` is null or provider is unavailable.
  Supports `${PR_NUM}` interpolation at runtime (via `sed` in the helper).

**Valid gate names:** `review`, `qa`, `docConsistency`. These are skills.json-specific
names, distinct from the profile gate names (`codeReview`, `qa`, `engReview`, etc.).
The mapping is:
- skills `review` → profile gate `codeReview`
- skills `qa` → profile gate `qa`
- skills `docConsistency` → hard-enforced (no profile gate — always runs)

### Preamble Helper: `rh_skill_command`

```bash
# Returns the skill suggestion text for a gate.
# If skills.json exists and has a configured skill, returns "Run <skill>".
# Otherwise returns the fallback text (with ${PR_NUM} interpolated).
# Caches the parsed skills.json in _RH_SKILLS_JSON (read once per hook invocation).
# Usage: rh_skill_command "codeReview" "$PR_NUM"
_RH_SKILLS_JSON=""
_RH_SKILLS_LOADED=""

rh_skill_command() {
  local gate="$1"
  local pr_num="${2:-}"
  # Load and cache skills.json on first call
  if [ -z "$_RH_SKILLS_LOADED" ]; then
    _RH_SKILLS_JSON=$(cat .right-hooks/skills.json 2>/dev/null || echo "{}")
    _RH_SKILLS_LOADED=1
  fi
  # Use jq --arg to prevent injection (gate is a string key, not a jq path)
  local skill provider
  skill=$(echo "$_RH_SKILLS_JSON" | jq -r --arg g "$gate" '.[$g].skill // empty' 2>/dev/null)
  provider=$(echo "$_RH_SKILLS_JSON" | jq -r --arg g "$gate" '.[$g].provider // empty' 2>/dev/null)

  # Tier 1: Configured skill with available provider
  if [ -n "$skill" ]; then
    local available=true
    case "$provider" in
      gstack)      rh_has_gstack      || available=false ;;
      superpowers) rh_has_superpowers  || available=false ;;
    esac
    if [ "$available" = "true" ]; then
      echo "Run ${skill}"
      return
    fi
  fi

  # Tier 2: Fallback text from skills.json
  local fallback
  fallback=$(echo "$_RH_SKILLS_JSON" | jq -r --arg g "$gate" '.[$g].fallback // empty' 2>/dev/null)
  if [ -n "$fallback" ]; then
    echo "$fallback" | sed "s/\${PR_NUM}/${pr_num}/g"
    return
  fi

  # Tier 3: Runtime tool detection (backward compat — no skills.json)
  case "$gate" in
    codeReview)
      if rh_has_gstack; then echo "Run /review"; return; fi
      if rh_has_superpowers; then echo "Use superpowers:requesting-code-review"; return; fi
      ;;
    qa)
      if rh_has_gstack; then echo "Run /qa"; return; fi
      ;;
    docConsistency)
      if rh_has_gstack; then echo "Run /document-release"; return; fi
      ;;
  esac

  # Tier 4: Generic fallback
  echo "Post a comment on the PR"
}
```

**Return contract:** Always returns a non-empty string. Never fails. Order:
1. If `skills.json` exists and gate has non-null `skill` → `"Run <skill>"`
2. If gate has `fallback` → the fallback text (with `${PR_NUM}` interpolated)
3. Otherwise → generic `"Post a comment on the PR"`

**Security:** Uses `jq --arg` instead of string interpolation to prevent jq
injection from untrusted gate names.

**Caching:** `_RH_SKILLS_JSON` is loaded once per hook invocation (not per call).
Same pattern as existing `_RH_HAS_GSTACK` caching.

### Sentinel Protocol Hints (Expansion #4)

When hooks block for missing review/QA, the block message now includes the
sentinel file path. Before, agents had to discover this from CLAUDE.md.

```
🚫 stop-check — BLOCKED

• No review comment found. Run /review
  → Sentinel: write comment ID to .right-hooks/.review-comment-id

• No QA comment found. Run /qa
  → Sentinel: write comment ID to .right-hooks/.qa-comment-id
```

### Migration / Backward Compatibility

**Existing installations (no skills.json):** `rh_skill_command` returns generic
fallback text. Hooks behave exactly as before. No breakage.

**Init:** `init.js` generates `.right-hooks/skills.json` alongside `signatures.json`
using the same `detectTooling()` result. In `install()`, after the signatures copy
(~line 160):

```javascript
// Copy skills config (same detection logic as signatures)
const skillsSource = tooling.hasGstack ? 'skills-gstack.json'
  : tooling.hasSuperpowers ? 'skills-superpowers.json'
  : 'skills-generic.json';
const skillsSrc = path.join(signaturesDir, skillsSource);
const skillsDst = path.join(rhDir, 'skills.json');
if (fs.existsSync(skillsSrc)) {
  fs.copyFileSync(skillsSrc, skillsDst);
  console.log(`✓ Skills configured: ${skillsSource.replace('skills-', '').replace('.json', '')}`);
}
```

Existing `init` users who re-run get the file; those who don't are unaffected.

**Upgrade:** `upgrade.js` checks if `.right-hooks/skills.json` exists. If so, skip
entirely (always preserve — user config). If not, generate from current tooling
detection (same logic as init). In `run()`, after the rules upgrade:

```javascript
// Skills config — always preserve if exists, generate if missing
const skillsDst = path.join(rhDir, 'skills.json');
if (!fs.existsSync(skillsDst)) {
  // Generate default — same logic as init
}
```

### Doctor Skill Validation (Expansion #2)

Doctor adds these checks:
1. **skills.json exists** — warning if missing (not error — fallback works)
2. **Valid JSON** — error if malformed
3. **Provider installed** — for each gate where `provider` is set, check:
   - `"gstack"` → `rh_has_gstack()` equivalent (directory check)
   - `"superpowers"` → `rh_has_superpowers()` equivalent
   - `null` → skip (prompt-based, no provider needed)
   If provider not found → warning: "review skill /review requires gstack but
   gstack not found"

### `npx right-hooks skills` Status Command (Expansion #1)

```
🥊  Right Hooks — Skills

  Gate            Skill              Provider     Status
  review          /review            gstack       ✓ available
  qa              /qa                gstack       ✓ available
  docConsistency  /document-release  gstack       ✓ available
```

When provider is missing:
```
  review          /review            gstack       ⚠ gstack not found
```

When skill is null (prompt-based):
```
  review          (prompt-based)     —            ✓ fallback
```

### `npx right-hooks skills set` CLI (Expansion #3)

```bash
npx right-hooks skills set codeReview /my-custom-review
npx right-hooks skills set qa superpowers:requesting-code-review
npx right-hooks skills set docConsistency /document-release
```

Validates gate name against `["codeReview", "qa", "docConsistency"]`. Rejects
unknown gates with list of valid names. Reads existing skills.json, updates
the gate's `skill` field, and infers `provider` from the skill prefix:
- `/` prefix → `"gstack"` (gstack skills start with `/`)
- `superpowers:` prefix → `"superpowers"`
- Otherwise → `null` (custom/unknown provider)

Writes back using atomic write (write to `.right-hooks/.skills.json.tmp`,
rename to `skills.json`) to prevent corruption. The `fallback` field is not
updatable via CLI — users can edit JSON directly for custom fallback text.

## Scope

**In:** skills.json config, preamble helper, hook refactor (stop-check + pre-merge),
skills CLI (status + set), doctor validation, sentinel hints, learned-patterns dedup

**Out:**
- Plugin marketplace, auto-discovery, runtime plugin loading
- Per-branch-type skill variation, provider auto-installation
- `engReview` gate skill dispatch — eng review is a profile gate but has no
  standard skill across tools (gstack uses `/plan-eng-review` which is a plan-mode
  skill, not a post-implementation review). Can be added later if a pattern emerges.
- The superpowers `verification-before-completion` TIP in `stop-check.sh` line 106-108
  stays hardcoded — it's a behavioral hint, not a configurable skill dispatch
